# 陪玩项目改造记录

> 分支：`cursor/merge-xdxie-asr-tts-31b8`（相对 `main`）  
> 记录时间：2026-06-25  
> 目标：合并 xdxie/Game_Demo 的 ASR/TTS 能力，打通「麦克风 → ASR → VLM → TTS」全链路，并修复 mock / 实机（NitroGen fast_api）模式下的多项稳定性问题。  
> 约束：**未修改** `action_fast_system/` 目录。

---

## 一、整体架构（改造后）

```
前端 (app.js)
  ├─ 麦克风 PCM ──WebSocket 0x01──► ASRHandler ──► 用户话术
  ├─ 视频帧 JPEG ──WebSocket 0x02──► FrameBuffer ──► NitroGen / VLM
  └─ TTS MP3 ◄──WebSocket 0x03── TTSQueue ◄── VLM / 快通道

NitroGen 后端（三选一，由 .env 控制）：
  mock        → MockNitroGenClient（本地模拟，无 GPU）
  fast_api    → FastApiNitroGenClient（HTTP POST /predict，经 SSH 隧道）
  zmq         → NitroGenClient（旧 ZMQ 路径）

慢系统（用户说话后的回答）：
  ASR 转写 → VLMRequestManager → VLM（openai 网关 / anthropic / mock）→ TTS
```

## 二、按阶段汇总的改动

### 阶段 1：合并 xdxie ASR/TTS（commit `44ff0ef`）

**来源**：`https://github.com/xdxie/Game_Demo.git` 的 ASR/TTS 实现。

| 模块 | 改动 |
|------|------|
| `backend/asr/handler.py` | 支持 `faster-whisper` 与 `openai-whisper` 双引擎；自适应 VAD（短句/长句不同静音阈值）；静音裁剪；保留原有 barge-in、`seek_generation`、`asr_state`、`on_utterance` 回调 |
| `backend/tts/engine.py` | 支持 `volcengine` 与 `edge-tts` 双引擎；完整 MP3 送入 `TTSQueue`（非流式 chunk） |
| `backend/config.py` | 新增 `ASR_ENGINE`、`ASR_DEVICE`、`TTS_ENGINE`、`VOLC_*` 等配置项 |
| `backend/warmup.py` | 预热时按引擎类型加载 Whisper / TTS |
| `requirements.txt` | 补充 faster-whisper、volcengine 等依赖 |
| `.env.example` | 补充 ASR/TTS 引擎与环境变量说明 |
| `tests/test_tts_asr.py` | 新增交互式 ASR/TTS 联调脚本（不硬编码 API Key） |

---

### 阶段 2：修复麦克风不工作（commit `539012e`）

**现象**：日志只有 `app.js 304`、`favicon 404`，无 PCM 数据。

| 模块 | 改动 |
|------|------|
| `frontend/app.js` | 在「开始分析」点击时**先于**长时间 `await` 调用 `primeMicrophoneOnUserGesture()`，避免 Windows/Chrome 下 AudioContext 被挂起；拆分 mic 申请与 worklet 挂载；增加 PCM 发送日志 |
| `frontend/index.html` | `app.js` 加 cache-bust 版本号 |
| `backend/main.py` | WebSocket 关闭时打日志；无 session 时收到 PCM 打警告；`/favicon.ico` 返回 204 消除 404 |

---

### 阶段 3：ASR 空结果 / WebSocket 断开 / 性能（commit `bb40b2d`）

**现象**：NitroGen 慢、ASR 状态卡住、ASR 结果为空、WebSocket 重连。

| 模块 | 改动 |
|------|------|
| `backend/main.py` | 会话运行期间推迟 action timeline 批处理，避免与实时 NitroGen 抢资源；JPEG 解码放到线程池，不阻塞 WS 心跳；延长 `ws_ping_timeout` |
| `backend/asr/handler.py` | 空识别结果时记录音频时长/峰值；faster-whisper 启用 `vad_filter` |
| `frontend/app.js` | 记录 WebSocket 关闭 code；NitroGen 等待态 tooltip |

---

### 阶段 4：Barge-in 后 ASR 卡死（commit `5846cf5`）

**现象**：用户打断 TTS 后再次说话无反应。

**根因**：barge-in 后 `_barge_in_armed=False` 且 `_muted=True`，后续 PCM 全部被丢弃。

| 模块 | 改动 |
|------|------|
| `backend/asr/handler.py` | 仅在实际处理打断后才 disarm barge-in；TTS 已停但仍 muted 时自动 unmute；barge-in 成功后同一块 PCM 继续走 VAD；`force_unmute` 重置 VAD 并重新 arm barge-in；打断后 drain TTS 队列 |

---

### 阶段 5：视频结束 / mock 模式 / 快通道 TTS（commit `667582f`）

**现象**：mock=1 时视频结束无法说话；mock=0 卡在「说话中」；快 TTS 无反应。

| 模块 | 改动 |
|------|------|
| `backend/main.py` | 视频结束不 `frame_buffer.pause()`，保留最后一帧供语音问答；`on_video_ended` 广播可继续提问 |
| `backend/asr/handler.py` | `vad_max_speech_sec=8` 超时强制 flush，防止游戏底噪导致 VAD 永不结束 |
| `backend/nitrogen/mock_client.py` | mock 发出 `DODGE` / `is_action_change` 等信号 |
| `backend/fast/action_filter.py` | 对 `is_action_change` 触发快通道提示 |
| `backend/config.py` / `.env.example` | mock 下默认保留 `FAST_TTS=1` |
| `frontend/app.js` | 视频结束后重新 arm 麦克风 |

---

### 阶段 6：实机模式 NitroGen 不亮 / 说话无回应（commit `7807187`）

**现象**：`NITROGEN_MOCK=0`、`NITROGEN_BACKEND=fast_api`、SSH 隧道开启，绿点不亮，说话无 AI 回应。

| 模块 | 改动 |
|------|------|
| `backend/video/frame_buffer.py` | `seek()` 不再清空 `latest_frame`（避免 seek 后 VLM 无画面） |
| `frontend/app.js` | 首次 WS 连接不再发送破坏性 `seek`；`captureSnapshotFrame()` 在视频暂停时也可推帧 |
| `backend/main.py` | 无画面时用户提问广播 `request_frame`；`/probe/health` 与 `/session/status` 增加 `video_frames`、`has_video_frame` |
| `backend/nitrogen/fast_api_client.py` | 连接失败日志附带 URL 提示 |

---

### 阶段 7：NitroGen 健康探针与 UI 报错（commit `fe6f2c1`，最新）

**现象**：端口已开但远端 FastAPI 未就绪时，用户看不到具体原因。

| 模块 | 改动 |
|------|------|
| `backend/nitrogen/health.py` | **新增** `check_fast_api_health()`：检测端口 → `POST /reset` → `POST /predict` |
| `backend/main.py` | **新增** `GET /nitrogen/health`；会话启动时后台探针；广播 `nitrogen_state`；分析循环在视频暂停时仍更新 perception UI；用户提问前等待 JPEG 异步解码（最多 0.5s）；VLM 失败广播 `vlm_error` |
| `backend/nitrogen/fast_api_client.py` | 增加 `last_error`、`_last_ok_time` 状态 |
| `backend/slow/trigger.py` | VLM 用户问答失败时 `on_user_error` 回调 |
| `frontend/app.js` | 处理 `nitrogen_state`（绿/黄/红）；启动时展示 `nitrogen_health` 失败；VLM 错误系统消息；去重 NitroGen 重复报错 |
| `frontend/style.css` | `.dot.error` 红色状态 |
| `tests/test_nitrogen_health.py` | **新增** 健康探针单元测试 |

## 四、新增 / 变更的 HTTP 与 WebSocket 接口

### HTTP

| 路径 | 说明 |
|------|------|
| `GET /nitrogen/health` | NitroGen FastAPI 探针（端口 + /reset + /predict） |
| `GET /probe/health` | 增加 `video_frames`、`has_video_frame`、`nitrogen_health` |
| `GET /session/status` | 增加 `video_frames`、`has_video_frame` |
| `POST /start` | 返回 `nitrogen_health`（fast_api 模式） |
| `GET /favicon.ico` | 返回 204 |

### WebSocket（服务端 → 客户端，新增或增强）

| type | 说明 |
|------|------|
| `nitrogen_state` | NitroGen 连接/推理状态（ok / waiting / error） |
| `request_frame` | 后端缺画面时请求前端推一帧 |
| `status: user_question_no_frame` | 用户提问时无视频帧 |
| `status: video_ended_can_ask` | 视频结束仍可提问 |
| `status: vlm_error` | VLM 调用失败原因 |

## 五、环境变量（实机模式完整示例）

```env
# VLM（说话有回应必需）
VLM_MOCK=0
VLM_API_KEY=your-api-key
VLM_API_BASE=https://yunwu.ai/v1
VLM_MODEL=gemini-3.1-flash-lite:stable

# NitroGen 实机（绿点必需）
NITROGEN_MOCK=0
NITROGEN_BACKEND=fast_api
NITROGEN_FAST_API_URL=http://localhost:18000   # 勿用 8000（陪玩服务端口）
NITROGEN_SSH_TUNNEL=1
NITROGEN_SSH_HOST=connect.bjb1.seetacloud.com
NITROGEN_SSH_PORT=18037
NITROGEN_SSH_USER=root
NITROGEN_SSH_PASSWORD=your-ssh-password
NITROGEN_SSH_REMOTE_PORT=8000

# 可选
FAST_TTS=1
ASR_ENGINE=faster-whisper
TTS_ENGINE=edge-tts
```

## 六、排查清单（实机模式）

1. `http://localhost:8000/nitrogen/health` → `ok: true`
2. `http://localhost:8000/probe/health` → `video_frames > 0`，`nitrogen.inference_count > 0`
3. 聊天区无 `VLM mock` / `VLM 调用失败` / `NitroGen：…` 红色提示
4. 视频在播放（或至少有一帧 snapshot）
5. 远端 GPU 上 NitroGen FastAPI 已启动（见 `action_fast_system/README.md`）
